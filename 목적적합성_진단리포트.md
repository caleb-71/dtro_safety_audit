# DTRO 자체종합안전심사 시스템 — 목적 적합성 진단

진단 일자: 2026-07-02 | 기준: 사용자가 정의한 시스템 목적 4가지

## 진단 요약표

| # | 시스템 목적 | 구현 상태 | 판정 |
|---|---|---|---|
| 1 | 지적이 많은 분야·부서 분석 | 통계분석 + 부서별 조회 화면 완비 | ✅ 충족 |
| 2 | 지적사항 조치(이행실태) 분석 | 추진실적 데이터를 벡터 임베딩에만 사용, 통계 분석 없음 | ❌ **최대 갭** |
| 3 | 지적 감소를 위한 컨설팅 도출 | 규칙 기반 점검 목록만 제시, 개선방향 서술 없음 | ⚠️ 절반 |
| 4 | 심사 결과 보고서 작성 | 서식·표·차트는 우수, 서술형 분석문 부재 | ⚠️ 절반 |

## 목적 1. 지적 많은 분야·부서 분석 — ✅ 잘 구현됨

- `page_analysis.py`: 파트별 현황, 연도별 트렌드, 리스크 분석, 반복 지적 TOP10, 부서별 TOP10 모두 구현
- `page_dept.py`(README에 없는 신규 화면): 부서 선택 → 필터 → KPI → 반복 3회 경고 배너 → 유사사례·법령근거·체크포인트까지. **실무 심사 준비 화면으로 완성도 높음**

발견된 문제:
- `page_dept.py:530` "체크 결과 저장" 버튼이 실제로는 아무것도 저장하지 않고 성공 메시지만 표시 (허위 기능 — 저장 구현 또는 버튼 제거 필요)

## 목적 2. 조치 이행실태 분석 — ❌ 사실상 미구현 (최대 갭)

`추진실적(action_result)` 컬럼은 데이터에 존재하고 벡터DB 임베딩 텍스트에는 포함되지만, **이행실태를 분석하는 기능이 시스템 어디에도 없습니다**:

- 조치 완료율(전체/부서별/연도별) 통계 없음
- 미조치·형식적 조치("조치완료" 한 줄) 식별 없음
- 시정명령 이행 여부 추적 없음 (page_dept 경고 배너가 "이행 여부 확인!"이라고 안내만 함)
- 보고서에 이행실태 장(章) 없음

**권장 구현** (효과 대비 난이도 낮음):
1. 전처리 단계에서 `조치상태` 파생 컬럼 생성 — LLM으로 추진실적 텍스트를 `완료/진행중/미흡/형식적` 4단계 판정 (llama3.1:8b로 충분한 단순 분류 작업)
2. 부서별 이행률 = 완료건 ÷ 지적건 → page_dept·page_analysis에 추가
3. "반복 지적 + 이행률 낮음" 교차 부서 = 컨설팅 최우선 대상 자동 도출

## 목적 3. 컨설팅 방향 도출 — ⚠️ 절반 구현

현재: 보고서 6장 "집중 점검 권고"와 page_dept 체크포인트가 있지만, 내용이 **리스크 상 목록 + 반복 지적 목록의 나열**입니다. "왜 반복되는지, 어떤 개선 활동을 컨설팅해야 하는지"의 서술이 없습니다. 이 부분에 LLM이 전혀 사용되지 않습니다.

**권장 구현**: Safety-On의 예측 엔진과 동일한 패턴 적용 —
"부서 X: 반복 지적 A(4회), 이행률 60%, 유사사례 3건" 통계를 근거로 LLM에게
"반복 원인 추정 → 컨설팅 포인트 2~3개 → 차기 심사 확인사항" 구조의 서술 생성.
(근거 수치 인용 강제 규칙 포함 — Safety-On에서 검증된 방식)

## 목적 4. 심사 결과 보고서 — ⚠️ 형식은 우수, 내용이 절반

`report_builder.py`는 표지·목차·6개 장·표 서식·차트 삽입까지 잘 만들어져 있습니다. 그러나:

1. **서술형 분석 문장이 없음** — 모든 장이 표+건수 나열. "안전보건 파트가 전년 대비 증가한 원인은…" 같은 분석 문단이 없어 실무 보고서로 쓰려면 사람이 서술을 전부 추가해야 함. **LLM이 보고서 생성에 전혀 사용되지 않음** (README의 "AI 분석 보고서"라는 이름과 불일치)
2. **"2025년" 하드코딩** — `report_builder.py:86,121` 표지·파일명에 연도가 고정. 2026년 심사에 쓰면 잘못된 제목
3. **부서별 분석 장 없음** — 목적 1의 핵심인데 보고서에는 미반영
4. **이행실태 장 없음** — 목적 2와 동일 갭

## 그 외 발견된 버그

**🔴 반복 지적 리스크 상향 규칙이 죽어있음**
`classifier.py`의 리스크 판정에 "반복 3회 이상 → 상" 규칙이 있지만, 전체 분류를 수행하는 `rag_engine.classify_with_rag()`가 `repeat_count`를 계산해서 넘기지 않아 **항상 기본값 1로 판정**됩니다. 반복 지적사항의 리스크가 실제보다 낮게 매겨지고 있습니다.
→ 수정: `run_full_classification()`에서 제목별 빈도를 미리 계산해 전달 (5줄 수정)

**🟡 AI 질의응답이 이행실태 질문을 처리 못함**
`page_chat.py._extract_relevant_data()`가 부서·파트·반복·리스크·시정명령 키워드만 감지. "조치가 안 된 지적사항은?", "이행률 낮은 부서는?" 같은 질문에 관련 데이터가 컨텍스트에 들어가지 않음.

**🟢 잘된 점 (Safety-On보다 앞선 부분)**
- 대화 히스토리 파일 영구 저장 + 최근 5턴 컨텍스트 전달 (대화 이어가기 이미 구현됨)
- 3단계 분류(법령DB → 키워드 → LLM)의 단계적 설계와 유사도 임계값(0.55) 방어
- 분류 결과에 method/confidence 기록 — 사후 검증 가능한 구조

| 순위 | 작업 | 효과 | 규모 |
